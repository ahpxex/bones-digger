# 骨鉴 · OSTEOGNOSIS

> 中国社会科学院考古研究所合作项目 · 动物骨骼智能鉴定系统

---

## 一句话介绍

骨鉴是一套面向田野考古与实验室场景的动物骨骼 AI 鉴定系统 —— 一张照片，几秒钟
就能给出物种、骨位、置信度，以及每一条证据都对得上专家特征的可审计鉴定报告。

---

## 为什么做这个

动物考古学里，种属与骨位鉴定是所有研究的第一步：判错了种属，后续对遗址生业、
驯化史、礼制结构的所有讨论都会偏。但眼下这件事的瓶颈是人：全国能独立做动物骨
骼鉴定的专家不足百人，一个中等规模遗址几千件骨片，往往要等半年才能排到手。
田野一线的考古工作者拍到照片只能先堆着，影响发掘判断的时效性。

我们把这件事用 AI 做出来了，但没把它做成黑盒：每一个判定都能告诉你「我是从哪
条专家知识得出的结论」。这样专家审核时不再要逐个重鉴，而是对着证据链点头或驳
回即可 —— 把专家从机械劳动里释放出来。

## 核心亮点

### 1. 可审计的 AI 鉴定 —— 每一个结论都能被反驳

把 AI 引入考古这件事，已经有不少团队在做，但大部分最终都没能跑通最后一公
里 —— 因为产出的是一个黑盒：输入一张图片、输出一个标签、给一个置信度，专家
既没法认可，也没法质疑，只能当作「参考」，于是实际工作里又退回到从头人眼鉴
定，AI 成了锦上添花的装饰。

我们从立项第一天就想通了这件事：**考古鉴定不是分类任务，而是一个论证任务**。
专家做一次鉴定，产出的其实不是「这是马的股骨」这句话，而是一段推导过程 ——
哪里看到了什么、对应到哪条经验、由此排除哪些竞品、最终收敛到哪个判定。这段
推导过程才是真正有学术价值、可以被继承与复核的东西。

所以我们把 AI 的输出也设计成了这种论证链结构。详情页中央最核心的那块「推理
依据链」每条证据都摊开成三列：左列是模型从图像里观察到的特征（例如「骨体呈
直形，无明显弯曲，两侧对称发育关节面」），中列是这条观察所对应的专家知识库
原文（例如「羊·胫骨：骨体直；骨壁光滑细腻；有髁关节面」），右列是这条证据
在最终判定里所占的权重百分比。三列拼在一起，就是一条可以被挨条质询、挨条推
翻、挨条溯源的明线论证。

再配合上面的六维度雷达（整体形态 / 关键结构 / 截面断面 / 表面纹理 / 表面痕迹 /
尺寸比例）与置信度候选排序 Top-3，评委拿到一份鉴定结论时，**不仅能看到模型
的答案，还能立刻判断模型是不是在瞎猜**。哪怕模型判错，审核者也能一眼定位到
哪条观察有偏差、哪条专家特征没匹配上，而不需要从零重新鉴定。从专家的角度，
这把「复核一次鉴定」的成本从过去的 1 小时压到了几分钟。

### 2. 专家知识硬约束 —— 杜绝大模型的一本正经胡说八道

多模态大模型最出名的缺陷就是「看图说话」时的幻觉：给它一张猪的下颌，它可能
非常自信地告诉你这是狗，并且编一套自洽的形态解释出来。这在消费场景里无非是
有趣，但在考古鉴定里就是灾难 —— 一条错误的判定会顺着简报、论文、考古报告一
路传下去。

我们解决这个问题的思路不是调 prompt，不是上 finetune，而是**直接从系统设计
层面把模型的自由度锁死**：

- 我们写了两部专家知识库。一部是 74 条由动物考古实验室整理的形态鉴定要点，
  按 7 物种 × 18 骨位组成交叉表 —— 每条要点是独立、可引用、编号唯一的原子知
  识。另一部是 14 条古代生态型六维档案，给出蒙古型黄牛、华南瘤牛、汗血马、
  果下马、骡等古代品种的身世、遗址、食谱、职业、病理、礼制等文化语境。两部
  知识库全部挂在知识图典页面上，评委、专家、田野队员可以随时翻阅核对 ——
  **我们不藏知识，我们公开课本**。
- 每次鉴定前，系统先用 Qwen3-Embedding 把查询和知识库全部向量化，做一次语义
  相似度检索，取出最相关的若干条；这些条目作为硬约束塞进多模态模型的 prompt
  中，并强制要求「你给出的每一条证据，必须显式引用上面提供的专家条目之一」。
- 模型的输出用严格 schema 做结构化校验，任何证据字段如果在知识库里找不到对
  应 ID 就会被直接丢弃；如果模型反复拿不出合法证据，系统会把结论降级为
  「未知 · 建议送专家」，而不是硬凑一个好看的答案糊过去。

这套「检索增强 + 硬约束 schema + 未知兜底」的三层防线，让骨鉴的输出始终被锚
定在真实专家知识上，不会凭空创造鉴定依据。用评委能理解的话说：**我们让模型
只能说它真正读过的话**。

### 3. 双通道推理 —— 兼顾田野现场的秒级响应与实验室的深度复核

考古场景有一个矛盾：田野发掘现场需要秒级响应 —— 队员蹲在探方边上拍了一张
照，希望立刻知道这个层位里挖出来的是什么，好决定下一步怎么发掘。但学术复核
环节又要求能承受深度推理 —— 一具有争议的骨片可能要让模型从多个角度反复论
证、对比多个候选。这两种诉求一条 prompt 搞不定。

骨鉴同时接入了 Qwen3.5 家族里的两位：

- **实时通道** 使用 Qwen3.5-Flash 稠密模型，延迟优化，平均 2 秒返回结论。整
  套 SAM 分割 + 向量检索 + 多模态推理 + 3D 重建的端到端走下来约 10 秒，足以
  让田野队员在扫码拍照的间隙内拿到初步判定。
- **精推通道** 使用 Qwen3.5-Plus Thinking MoE 旗舰模型，当实时通道的置信度
  低于阈值、候选中存在跨物种近似（例如羊 / 鹿胫骨极难区分的样本）、或样本被
  标记为疑似 OOD 时自动升级。精推通道会输出完整的多步思维链，把模型是怎么
  从几个候选里一步步推到最终结论的整个推理过程暴露给专家复核。

关键设计：**两通道的结论同时保留**，不会被一个覆盖另一个。详情页在需要时会
并列展示两条思维链，让审核者看到实时通道与精推通道是否一致、如果不一致那么
分歧点在哪。这把「快速 + 可靠」这两个在工程上通常冲突的诉求，通过通道分离
的方式变成了互补关系。

### 4. 从照片到数字标本 —— 一次鉴定产出的是一份可归档资产

每一次鉴定在后台还会自动走一次基于 Meta 最新开源的 SAM 3D Objects 管线：以
用户上传的同一张骨骼照片为输入，在 5–30 秒内完成单图三维重建，产出标准 .glb
网格，落盘并在浏览器端用 three.js 实时渲染为可拖拽、可缩放、可旋转的 3D
标本。

这一步把骨鉴从一个「鉴定工具」升级为了「数字馆藏系统」。一次鉴定完成之后：

- **一张原片** —— 留作图像档案
- **一张分割图** —— SAM 3.1 圈出的主体，主体边界清晰，便于后续复检
- **一份结构化鉴定 JSON** —— 物种、骨位、置信度、六维评分、证据链、思维链
  全部可机读
- **一份 A4 排版的正式鉴定报告** —— 可打印、可归档
- **一个 3D 数字标本** —— 标准 glTF / .glb 格式，可被任何主流考古 / 博物馆
  / 游戏引擎直接复用

**这五项产物合起来，才是一次真正意义上的「数字化考古鉴定」**。其中 3D 数字
标本这一项特别有长期价值 —— 随着鉴定记录不断累积，骨鉴自然就长成了一座数
字标本馆，未来接入中国考古地理信息平台或开放数据仓库都只是一个接口的工作。

### 5. 著录与报告 —— 贴合考古工作流的成品形态

考古是一个高度讲究「著录」的学科：每一件出土物都要有唯一编号、出土背景、层
位信息、鉴定记录，这些信息最终沉淀为考古报告的骨骼部分。骨鉴在产品形态上就
是按这个工作流做的：

- **著录堂** 是所有历次鉴定的自动归档中心，按时间倒序组织缩略图 + 种属·骨位
  + 置信度 + 唯一编号，既是操作历史，也是资产清单。
- **鉴定报告** 是一份 A4 规格、Songti 宋体排版、朱砂印章落款的正式文书，
  Ctrl / ⌘ + P 一键导出 PDF；报告六章（结论 / 影像 / 排序 / 评分 / 证据 /
  思维链）的结构本身就贴合考古简报中「标本鉴定」章节的体例。

这意味着骨鉴**不只是一个演示级 demo，而是一个可以直接插入真实考古项目的工作
组件**：田野队员在现场扫描入库，鉴定结果随手转入考古报告，不需要任何额外加
工。

### 6. 传统美学 —— 不是装饰，而是学术语言的延续

整站采用朱砂红 · 铜褐 · 米黄纸面 · 宋体的视觉系统，章节序用汉字一至七 / 壹
至柒，分割线、角饰、印章、卷轴等元素全部自绘，不引入第三方 UI 库以避免与传
统美学的语汇冲突。

这是一个有意识的选择。考古学是一个传统极深的学科，从青铜器铭文到敦煌写卷、
从殷墟甲骨到宋版典籍，**学术报告的视觉语言本身就是学科身份认同的一部分**。
如果我们把骨鉴做成一个彩色渐变、圆角卡片、emoji 点缀的 SaaS 感界面，它也许
技术上没问题，但永远没法被真正的考古工作者当成自己人。

我们选择了一条更难走、但对的路：让 AI 用考古学自己的语言说话。所以导出的鉴
定报告看起来像考古简报，印章、汉字序、朱砂竖条让整个产品散发出典籍的质感
—— 这不只是美化，这是一种学科尊重，也是让产品真正落地到学术社区的前提。

## 技术栈

本项目采用了一套相对克制但都为核心场景量身定制的技术栈，每个选型背后都有明
确的工程动机，没有为了技术噱头而堆料的部分。

### AI 核心管线

- **多模态大模型**：阿里云百炼 Qwen3.5-Flash（实时通道）+ Qwen3.5-Plus Thinking
  （精推通道）。Qwen3.5 系列是 2026 年初发布的原生多模态模型，在中文场景、
  专业术语理解、结构化输出稳定性上明显优于前代；Flash 的稠密架构做低延迟优
  化，Plus 的 MoE 旗舰适合复杂推理，两者搭配覆盖了田野秒级响应到实验室深度
  复核的全部诉求。
- **图像分割**：Meta Segment Anything Model 3.1（SAM 3.1）。用带文本提示的
  开放词表分割能力，把骨骼主体从泥土、比例尺、手指等干扰中精准切出，作为后
  续多模态推理与三维重建的统一输入。
- **单图三维重建**：Meta SAM 3D Objects。2026 年 3 月 Meta 开源的单图 3D 重
  建模型，在单张 2D 骨骼照片上直接产出带纹理的 glTF 网格，完全替代了此前项
  目早期的 3DGS + COLMAP + splatfacto 多图管线 —— 那条路径要求用户拍 20
  张以上，SAM 3D 只需一张。
- **向量检索**：阿里云百炼 Qwen3-Embedding（text-embedding-v4）。1024 维向
  量，对中文术语与长描述的语义匹配效果显著优于传统词袋；我们把 74 条专家形
  态条目在服务端冷启动时全量编码、缓存进内存，查询时余弦排序 Top-K，端到端
  延迟 100ms 级。
- **兜底检索**：自研 bigram 二元组 + 对数长度惩罚算法，当云端 Embedding 服
  务不可达或 key 未配置时无缝接管，保证系统在完全离线的田野环境下也能出结
  果。

### 前端与渲染

- **Web 框架**：Next.js 16 + React 19，使用 Turbopack 作为打包器。Next.js
  16 的 React Server Components 让详情页与列表页的首屏直接由服务端渲染完
  整 HTML，对田野现场可能很差的移动网络非常友好；Server Actions 让上传 →
  鉴定 → 跳转这条核心链路完全不需要前端写 API 胶水层代码。
- **3D 实时渲染**：@react-three/fiber + @react-three/drei + three.js。前者
  让 Three.js 场景以 React 组件方式组织，后者提供 OrbitControls、useGLTF、
  Bounds 等工具性组件，三方配合让 3D 标本 viewer 既能加载真实 .glb 网格，也
  能在未接入重建服务时回落到我们用 Three.js 原生几何体程序化生成的长骨占位
  模型。
- **图表可视化**：Apache ECharts（SVG 渲染模式）。六维度雷达图用 SVG 输出，
  保证打印机与 PDF 导出时矢量清晰、文字可复制；同时 SVG 下的文件体积远小于
  Canvas 截图。
- **图像处理**：sharp（libvips 的 Node 绑定）。所有上传图片在服务端做 EXIF
  自动旋转、等比缩放到 1600px 以内、mozjpeg 90 质量重编码，既控制带宽也为
  后续 SAM 3.1 与 SAM 3D 提供统一尺寸输入。
- **TypeScript + Zod**：全链路严格类型 + 运行时 schema 校验。特别是大模型返
  回的结构化 JSON，用 Zod 拒绝任何不符合 schema 的字段，这是让「硬约束」能
  真正落地的关键工程机制。

### 部署与运行

- **开发环境**：Bun + Turbopack，冷启动不到 1 秒，改完立刻重载。
- **生产部署**：标准 Next.js 构建，可部署到任何支持 Node.js 的主机，包括
  Vercel、自建 Docker、以及考古研究所内网服务器。
- **数据持久化**：目前采用文件系统（JSON + 图像 + .glb）直接落盘的极简方
  案，便于评委阅卷与下阶段接入真实考古数据库。所有鉴定记录与资产文件一一对
  应、可原子删除、不留孤儿文件。

### 为什么不用这些

顺便说明一下我们**没有**选用的东西，这些也是经过权衡的：

- 没有用 LangChain / LlamaIndex 等 RAG 框架 —— 74 条结构化条目的检索逻辑自
  己写清楚不到 100 行，引入框架只会增加依赖与心智负担。
- 没有用 Pinecone / Weaviate 等向量数据库 —— 知识库只有 74 条，全量内存余弦
  即可，P99 延迟比任何外部服务都低。
- 没有用 shadcn / MUI / Ant Design 等 UI 库 —— 传统美学的版式语言与通用组件
  库的视觉语汇不兼容，直接用 Tailwind + 自写 `cer-*` 工具类反而更简单可控。
- 没有用 Redux / Zustand 等状态库 —— 整站是 RSC 为主，客户端状态只在上传组
  件里有几个 `useState`，够用就好。
